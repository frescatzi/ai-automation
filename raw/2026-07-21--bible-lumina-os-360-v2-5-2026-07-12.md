## A. Résumé exécutif 360°

LUMINA OS = écosystème d'automatisation & d'agents IA sur **n8n auto-hébergé** (`n8n.aftersunpeople.com`, Hetzner + Coolify), **brand-agnostic** : *1 socle, N marques*. Marque pilote : AFTER SUN PEOPLE (AFTRSN).

```
        📥 01-INTAKE            🧠 02-MEMORY                 🤖 03-BRAIN                    🔄 04-AUTOMATION   🔌 05-CONNECT
  Drive "Lumina Inbox"    GitHub wiki ──▶ pgvector      AFTRSN-Maestro (hub)          GitHub push ──▶      LUMINA-MCP-Server
  ──▶ classifieur LLM     (<brand>_memory) + Notion       ├─ 7 sous-agents Call        ORCHESTRATOR-SYNC    (search_brand_memory
  ──▶ GitHub raw/         collections : canon ·           ├─ Knowledge (memory-search) (déclenche PGVECTOR   pour clients externes)
  + archivage raw──▶Drive episodic · insights ·           ├─ LUMINA-AI-Router 🧭       + NOTION)
                          docs · skills                    │   (multi-LLM + RAG + PII)
                          écriture épisodique +            └─ Save Episode
                          consolidation nocturne          LUMINA-Hermes-Exec ⚙️
                                                          (exécuteur apprenant, skills)
                                        🛠️ 08-UTIL : consoles SQL / DDL (manuelles) · ⚫ Z_ARCHIVES · 🔵 SandBox
```

**Les 3 piliers :** (1) **Mémoire** — banques pgvector par marque (`<code>_memory`) + partagée (`lumina_memory`), alimentées Drive→GitHub (source de vérité)→pgvector/Notion ; mémoire vivante (épisodes→insights la nuit). (2) **Cerveau** — Maestro orchestre 7 sous-agents spécialisés ; le Router choisit le LLM par `task_type` avec garde-fou PII et injection RAG ; Hermes exécute en apprenant (bibliothèque de skills). (3) **Chaîne de traitement** — workflows rangés en dossiers-étapes numérotés 01→05 + 08.

## B. Glossaire

| Terme | Définition |
|---|---|
| **Lumina OS** | Le socle partagé (workflows 🌐 SHARED) : intake, mémoire, router, exécuteur, MCP, utilitaires. |
| **Marque** | Une entité cliente du socle (ex. AFTRSN). Workflows spécifiques taggés (ex. 🌞 AFTRSN). |
| **Banque `<code>_memory`** | Table pgvector d'une marque (ex. `aftrsn_memory`, `lumina_memory`). Schéma : id, title, content, collection, knowledge_type, source, source_ref, content_hash UNIQUE, embedding vector(1536), created_at, updated_at. Index HNSW cosinus. |
| **Collection** | Sous-ensemble d'une banque : `canon` (vérités de marque), `episodic` (épisodes bruts des agents), `insights` (enseignements consolidés), `docs` (wiki), `skills` (procédures apprises par Hermes). |
| **Agent** | Workflow n8n avec node AI Agent + LLM + Chat Memory + outils. |
| **Sous-agent** | Agent spécialisé appelé par le Maestro via `Call '…'` (toolWorkflow → executeWorkflowTrigger ; n'exige PAS que le workflow soit publié). |
| **Routeur** | LUMINA-AI-Router : choisit le modèle LLM selon `task_type` via OpenRouter, bloque le PII, injecte le RAG. |
| **Hermes** | Agent Ops auto-hébergé (`hermes.aftersunpeople.com`, Coolify). Branché via LUMINA-Hermes-Exec (cookie-auth par credential Custom Auth). |
| **Skill** | Procédure réutilisable stockée en mémoire (collection `skills`), recherchée par similarité avant chaque exécution Hermes ; auto-promue par la consolidation nocturne. |
| **Canon** | La bible de marque ingérée (collection `canon`) — la vérité de la marque pour les agents. |
| **Épisode** | Trace d'une interaction agent, écrite par `Save Episode` (collection `episodic`). |
| **Insight** | Enseignement condensé la nuit à partir des épisodes (collection `insights`, dédup sémantique < 0.12). |
| **memory_registry / brand_profiles** | Tables de config multi-marques : registre des banques + profil (tone_voice, forbidden_words, kb_ref). |

## C. Les catégories (dossiers-étapes)

Standard **process-flow** : les numéros = ordre du flux de données. Mêmes dossiers dans chaque marque.

| Dossier | Rôle dans la chaîne | Où chercher si problème |
|---|---|---|
| `01-INTAKE` 📥 | Entrée des données brutes (Drive → GitHub, archivage) | Fichiers qui n'arrivent pas dans GitHub/le wiki ; Inbox qui déborde |
| `02-MEMORY` 🧠 | Connaissance : ingestion, recherche, écriture, consolidation, provisioning | Agents qui « ne savent pas » ; recherche vide ; doublons |
| `03-BRAIN` 🤖 | Agents (Maestro + `Sub-Agents`), Router 🧭, Hermes-Exec ⚙️ | Réponses incohérentes, mauvais modèle, délégation cassée |

| `05-CONNECT` 🔌 | Exposition vers l'extérieur (MCP) | Clients externes (Claude, etc.) sans accès mémoire |
| `08-UTIL` 🛠️ | Utilitaires manuels (SQL/DDL) | — (outils d'admin, jamais actifs) |
| `Z_ARCHIVES` ⚫ | Dépréciés (`ZZZ_<date>_…`) | Ne jamais réactiver sans revue |
| `SandBox` 🔵 | Essais jetables | Aucune garantie |

**Debug :** incident en **amont** (donnée absente/mauvaise) → regarder 01/02 ; incident en **aval** (réponse/sortie) → 03/04/05.
