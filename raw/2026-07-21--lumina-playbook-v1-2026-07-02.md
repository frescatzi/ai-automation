# LUMINA-PLAYBOOK v1 — Généralisation multi-marques
**Date : 2026-07-02.** Blueprint réutilisable : monter un système multi-agents pour une nouvelle marque par **config + adaptation**, pas reconstruction. AFTRSN = marque pilote ; validé bout-en-bout avec une marque démo.

## 1. Principe : 1 workflow, N marques

Le système est **brand-agnostic** : le code ne change pas d'une marque à l'autre. Ce qui distingue une marque = sa **banque mémoire** (`<code>_memory`), son **profil** (`brand_profiles`) et son **canon** ingéré. Tout le reste (Router, mémoire vivante, skills, agents) est partagé ou paramétré.

**Couches partagées (une seule fois) :**
- `LUMINA-AI-Router` (routage LLM par `task_type`, garde-fou PII, RAG).
- `lumina_memory` (mémoire process partagée) + skills génériques (`collection='skills'`).
- Mémoire vivante : `memory-search`, `LUMINA-MEMORY-WRITE`, consolidation nocturne, `skill_registry`.
- Hermes (exécuteur apprenant).

**Couches par marque (à créer) :**
- Banque `<code>_memory` (+ index HNSW/collection/knowledge_type).
- Ligne dans `memory_registry` + ligne dans `brand_profiles`.
- Canon (bible) ingéré dans `<code>_memory` (`collection='canon'`).
- Jeu d'agents pointant sur `brand=<code>` (cloner le roster AFTRSN).

## 2. Le registre de marque

- **`memory_registry`** : `code, table_name, display_name, kind, active`.
- **`brand_profiles`** : `code, brand_name, tone_voice, forbidden_words (text[]), kb_ref`.
- Résolution dynamique : les workflows dérivent la table = `<code>_memory` (sanitizé `[a-z0-9_]`). Ajouter une marque enregistrée = aucun edit de code.

## 3. Onboarding d'une nouvelle marque (checklist)

1. **Provisionner** : exécuter `LUMINA-BRAND-PROVISION` avec `{code, brand_name, tone_voice, forbidden_words, kb_ref}` → crée `<code>_memory` + index + `memory_registry` + `brand_profiles` en une fois.
2. **Ingérer le canon** : déposer la bible de marque (PDF/MD) → `memory-write` (ou pipeline Drive→bank) avec `brand=<code>`, `collection='canon'`.
3. **Cloner le roster** : dupliquer Maestro + sub-agents ; dans chaque agent, régler l'outil Knowledge sur `brand=<code>` (spécialistes) / `lumina`+`<code>` (Maestro). Adapter les prompts avec le `brand_profile` (nom, voix, mots interdits, naming rules).
4. **Vérifier** : `memory-search {brand:<code>}` renvoie le canon ; un test de délégation Maestro fonctionne.
5. **Skills** : les skills génériques (lumina) s'appliquent d'emblée ; les skills spécifiques s'accumulent via l'auto-promotion nocturne sur la banque de la marque.

*Validation v1 : marque `demo` provisionnée, canon écrit et relu via les workflows partagés, sans toucher au code.*

## 4. Roster template (à cloner par marque)

- **Maestro** — orchestrateur, point d'entrée/sortie unique, mémoire `lumina`+`<code>`.
- **Challengers** (mémoire `<code>`) : Culture/Community, Experience/Event — pressure-test, ne produisent pas de livrable final.
- **Fonctions** : Marketing (+ garde-voix de marque), Secrétaire (draft-only), Content (réseaux/newsletter).
- **Hermes** = outil Ops apprenant (skills), pas un agent.
- Collaboration : orchestré + appels ciblés ; **actions critiques toujours gatées par le propriétaire de la marque**.

## 5. Prompt template (paramétré par `brand_profile`)

Blocs communs à injecter dans chaque agent : identité + rôle ; `MEMORY: query Knowledge (brand=<code>) avant de répondre` ; `NAMING` (nom exact + mots interdits depuis `forbidden_words`) ; `BRAND VOICE` (depuis `tone_voice`) ; langue = celle de l'utilisateur ; `AUTHORITY` (propriétaire = décideur final, actions critiques gatées). Modèles par rôle via OpenRouter (`~…-latest`).

## 6. Garde-fous (invariants, toutes marques)

- `task_type='private'` (PII) → jamais au cloud.
- Actions critiques (dépense, envoi, publication, suppression) → validation humaine, quel que soit le niveau de maturité d'un skill.
- Embedding figé `text-embedding-3-small` 1536 partout.
- Git = source de vérité ; pgvector/Notion = dérivés régénérables.

## 7. Références (POS génériques = briques du playbook)
